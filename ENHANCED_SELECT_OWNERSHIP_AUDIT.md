# Enhanced Select Ownership Audit

Scope:
- `src/styles/app.css`
- `src/styles/core.css`
- `src/styles/reservations.css`
- `src/scripts/ui/enhancedSelect.js`
- related runtime entrypoints that initialize or refresh enhanced select

Out of scope:
- broad cleanup
- generic dark control refactor
- semantic/status styling
- export/template CSS
- design-language unification

## Executive Summary

The enhanced-select system is currently app-owned and should remain app-owned for now.

`src/styles/core.css` does not own or style enhanced-select selectors directly. The component is defined by:

- JS-generated markup and state in `src/scripts/ui/enhancedSelect.js`
- base, dark, and RTL styling in `src/styles/app.css`
- local layout/width overrides in reservations and equipment filter contexts

Architecturally, this is a component system rather than a shared primitive. It does depend on shared control and surface tokens from `core.css`, but not on `core.css` owning component selectors.

Recommendation:
- short term: keep enhanced-select in `app.css`
- medium term: extract later to a dedicated component stylesheet imported by `app.css`
- do not move it into `core.css`

Extraction is feasible later, but not worth doing as a blind cleanup today because the component still has local layout coupling in equipment and reservations flows.

## Ownership Recommendation

### Recommended owner now

- `src/styles/app.css`

Why:
- the component is runtime-specific and JS-driven
- the selector family is self-contained under `.enhanced-select*`
- dark behavior is app-owned via `:where(html.dark, body.dark)`
- `core.css` currently has zero direct selector ownership here

### Recommended long-term destination

- dedicated component stylesheet, imported by `src/styles/app.css`

Why not `core.css`:
- enhanced-select is not a generic primitive like native `input` or `select`
- it depends on custom DOM generated by `src/scripts/ui/enhancedSelect.js`
- local reservations/equipment layout rules should remain outside any shared primitive layer

### Legacy tolerated

- none significant inside the core component block

The nearby generic select system in `app.css` is a separate ownership problem, but it is not enhanced-select residue.

### Candidate for later extraction

- all component-owned `.enhanced-select*` selectors from `app.css`
- JS-driven state selectors that are self-contained
- dark component block that mirrors the component structure cleanly

### Unknown / needs manual review

- invalid/error styling expectations if future forms require visible error state
- z-index/overflow interactions in modal-heavy contexts
- whether any page-local wrapper assumptions exist outside equipment and reservations flows

## Coupling Risks

### Low-risk coupling

- dependency on shared tokens from `core.css`
- this is appropriate and should remain

### Medium-risk coupling

- component styling is currently adjacent to generic `form-select` styling in `app.css`
- this makes future separation slightly noisier, but still manageable

### Higher-risk coupling

- equipment filter sizing and mobile layout hooks in `app.css`
- reservations filter-bar sizing and dark overrides in `reservations.css`

These are not reasons to keep the component in `core.css`. They are reasons to split extraction into:

1. component-owned selectors
2. page/local layout selectors

## Extraction Feasibility

### Safe now?

- no

Reason:
- extraction would still need a deliberate split between component selectors and page/local layout hooks
- doing that now is architecture work, not a mechanical move

### Safe later?

- yes

Reason:
- the component selector family is self-contained
- `core.css` does not own it
- JS markup/state is centralized
- local reservations/equipment hooks can stay in local files while the base component moves out

### Not worth doing?

- no

Reason:
- long-term, a dedicated component stylesheet would improve ownership clarity
- it is a reasonable later migration once the global control-layer conflict is settled or intentionally frozen

## What Should Stay As-Is

- enhanced-select remains app-owned in `src/styles/app.css`
- `core.css` continues to supply tokens only
- reservations-specific width/layout hooks stay in `src/styles/reservations.css`
- equipment-specific width/layout hooks stay near equipment styling until a dedicated extraction batch is planned
- no move into `core.css`
- no generic control refactor mixed into this component decision
